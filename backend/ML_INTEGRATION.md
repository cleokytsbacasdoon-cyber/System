## Seed Models (Required for Fresh Install)

Two files are tracked in git and must always be present:

```
backend/models/tourism_xgb_model.json
backend/models/tourism_model_metadata.json
```

These are the base XGBoost model used before any retraining has occurred.

All other model files in `backend/models/` are dynamically generated by retraining and are **git-ignored** â€” they persist on disk but are not committed to the repository.

## Retraining Flow

### Automatic (Monthly Cron)

The backend runs a cron job each month. When triggered:

1. Checks whether actual tourist arrival data exists in `monthly_tourism_dataset` for the target month
2. If data exists: trains all four models, compares accuracy, activates the winner
3. If no data: retraining is skipped (no error, just deferred)

### Manual (via API or Dashboard)

**API:**
```
POST /api/ml/retrain/simulate-monthly
Body: { "baseModelName": "xgboost_base_march_2026", "year": 2026, "month": 4 }
```

**Dashboard:**
Go to **Model Parameters** tab â†’ Select month/year â†’ Click **Retrain Models**

Both paths enforce a zero-data guard: if `SUM(arrivals) = 0` for the selected month, a 400 error is returned with a clear message. This prevents training on empty months.

## compareAllModels Function

All retraining (manual and auto) runs through `compareAllModels(year, month, baseModelName, triggerType)`:

1. Runs all four models against the same dataset
2. Picks the winner by highest accuracy
3. Archives all currently active models (`UPDATE ... SET status = 'archived' WHERE status = 'active'`)
4. Sets the winner as active
5. Saves both historical and future predictions to `saved_predictions` table

---

## saved_predictions Table

After each retrain, predictions are upserted into `saved_predictions`:

- **Historical months** (have actual data): stores both actual and predicted, enabling error calculation
- **Future months** (no actual data yet): stores predicted only; uses `DO NOTHING` on conflict to preserve the first-ever forecast for each future month

## GET /api/forecasts Behavior

The `/api/forecasts` endpoint serves data to the dashboard charts:

1. If `saved_predictions` has rows â†’ returns them paired with actual arrivals from `monthly_tourism_dataset`
2. Future months not yet in `saved_predictions` â†’ appended from a live Python prediction call
3. If `saved_predictions` is empty â†’ full live Python prediction fallback

Force legacy mode (skips ML): `GET /api/forecasts?source=legacy`

Model-specific forecasts: `GET /api/forecasts?model=lstm` (options: `xgboost`, `lstm`, `random_forest`, `prophet`)

## Model Storage

- `MODEL_STORAGE_DIR` defaults to `backend/models/`
- Can be overridden via `ML_MODEL_DIR` environment variable
- Model file naming convention: `{type}_{base|eval}_{month}_{year}.json`
  - Example: `xgboost_eval_march_2026.json`, `lstm_base_december_2025.json`
- Metadata files: same name with `_metadata.json` suffix
